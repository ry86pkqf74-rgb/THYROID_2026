## Root Cause

`operative_episode_detail_v2` on MotherDuck was materialized from the base SQL
(script 22) before `enrich_from_v2_extractors()` ran. The SQL creates all 11 NLP
boolean fields as `FALSE` / `NULL` defaults; the Python enrichment step then UPDATEs
them from clinical note NLP. When script 26 materialized to MotherDuck, it copied
the pre-enrichment state.

## Fix Applied

A targeted script (`scripts/71_operative_nlp_to_motherduck.py`) was written to:

1. Connect to MotherDuck RW  
2. Load all 10,499 clinical notes from `clinical_notes_long`  
3. Run `OperativeDetailExtractor` (regex-based NLP) on each note  
4. Stage 4,469 enrichment rows in `_v2_operative_enrichment`  
5. UPDATE `operative_episode_detail_v2` in-place via the same DISTINCT ON / date-proximity
   matching used in script 22  
6. Recreate `md_oper_episode_detail_v2` mirror  
7. Drop staging table  

Total extraction: 13,186 operative entities from 10,499 notes in ~126 seconds.
